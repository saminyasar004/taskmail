# Agent Coding Rules

These rules apply to ALL AI agents and models working on this repository.
Read this file before writing any code. These are non-negotiable standards.

---

## 1. No Em Dashes

Never use em dashes (--) in code, comments, or strings.
Use a regular hyphen (-) or rewrite the sentence instead.

Bad:
```ts
// This service handles Gmail -- including watch setup
const label = "action required -- reply needed";
```

Good:
```ts
// This service handles Gmail, including watch setup
const label = "action required - reply needed";
```

---

## 2. Humanized Code

Write code like a thoughtful senior engineer wrote it, not a machine.

- Use clear, readable variable names. No single-letter variables except `i` in simple loops.
- Write comments that explain WHY, not WHAT.
- Break complex logic into named functions.
- Avoid deeply nested callbacks. Use async/await.
- Prefer descriptive error messages over generic ones.

Bad:
```ts
const d = await prisma.user.findUnique({ where: { id: uid } });
if (!d) return c.json({ error: "err" }, 400);
```

Good:
```ts
const user = await prisma.user.findUnique({ where: { id: userId } });
if (!user) return c.json({ error: "User not found. Please sign in again." }, 404);
```

---

## 3. Swagger Documentation is Mandatory

Every endpoint that gets added to the Hono backend MUST be documented in the OpenAPI spec.

- File location: `apps/api/src/lib/openapi.ts`
- The spec is served at `GET /api/docs/spec.json`
- The Swagger UI is served at `GET /api/docs`
- After adding a new route, always update the `paths` section of the spec

What each endpoint entry must include:
- `summary` - one-line description
- `description` - full explanation of what the endpoint does and when to use it
- `tags` - group endpoints logically (e.g., "Tasks", "Emails", "Briefing", "Gmail")
- `parameters` - all path, query, and header params with types and descriptions
- `requestBody` - full schema for POST/PATCH bodies
- `responses` - all possible response codes (200, 201, 400, 401, 404, 500) with example payloads

---

## 4. TypeScript Only

No JavaScript files in `apps/api/src` or `apps/web/src`.
All code must be typed. Avoid `any` unless absolutely unavoidable - and if you must use it, add a comment explaining why.

---

## 5. ESM Import Extensions

In `apps/api`, all relative imports must include `.js` extension because the package uses `"type": "module"`.

Bad:
```ts
import { prisma } from "./prisma";
```

Good:
```ts
import { prisma } from "./prisma.js";
```

This does NOT apply to `apps/web` (Next.js handles module resolution itself).

---

## 6. Environment Variables

- Never hardcode secrets, keys, or credentials in source code.
- Always read from `process.env`.
- If a required env variable is missing, log a clear error message and throw - do not silently fail.

---

## 7. Error Handling

Every async route handler must have a try/catch block.
Log errors with enough context to debug: include the user ID or entity ID where relevant.

Bad:
```ts
} catch (error) {
  return c.json({ error: "Something went wrong" }, 500);
}
```

Good:
```ts
} catch (error) {
  console.error(`Failed to fetch tasks for user ${user.id}:`, error);
  return c.json({ error: "Internal Server Error" }, 500);
}
```

---

## 8. No Unused Imports or Variables

Before committing, remove any unused imports or declared-but-never-used variables.
The TypeScript compiler should be clean with zero warnings.
